# Brainstorm Before Propose

> Source: [omo-openspec-tdd.md](../../omo-openspec-tdd.md) §3.3 | Inspired by [Superpowers brainstorming](https://github.com/obra/superpowers)

Transform rough feature ideas into fully-formed designs through structured Socratic dialogue,
BEFORE generating any OpenSpec artifacts.

**When to use**: EVERY time before `/opsx-propose`. No exceptions — even "simple" features benefit.
The goal is to ensure the spec generated by propose actually matches what the user wants.

<HARD-GATE>
Do NOT invoke `/opsx-propose`, write any spec files, or take any implementation action
until the design has been discussed and the user has approved it.
</HARD-GATE>

## Why This Exists

Without brainstorming, `/opsx-propose` generates specs based on surface-level descriptions.
The result: specs that miss edge cases, overlook constraints, or solve the wrong problem.
Five minutes of Socratic Q&A saves hours of rework.

## Workflow

```
/opsx-brainstorm <feature-name>
    │
    ├─ Phase 1: Context (read project, understand domain)
    ├─ Phase 2: Clarify (Socratic Q&A, one question at a time)
    ├─ Phase 3: Propose approaches (2-3 options with trade-offs)
    ├─ Phase 4: Present design (incremental, section by section)
    └─ Phase 5: Transition → /opsx-propose <feature-name>
```

## Phase 1: Understand Context

Before asking any questions:

1. Read `AGENTS.md` to understand project structure and conventions
2. Read relevant source files mentioned in the user's request
3. Check `openspec/specs/` for existing specs that relate to this feature
4. Understand the domain enough to ask informed questions

**Output**: Internal understanding. No user message yet — just preparation.

## Phase 2: Socratic Q&A

Ask questions **one at a time** to refine the idea. Each question should be specific and actionable.

### Question Types (in order of priority)

1. **Purpose**: "What problem does this solve? Who is the user?"
2. **Scope**: "Should this handle [edge case]? What's explicitly OUT of scope?"
3. **Constraints**: "Any performance requirements? Compatibility constraints? Timeline?"
4. **Success criteria**: "How do you know this is done? What does 'working' look like?"
5. **Preferences**: "Between [A] and [B], which do you prefer? Why?"

### Rules

- **One question per message**. Never ask 3+ questions at once.
- **Prefer multiple choice** when possible: "Should this be (A) real-time, (B) batch, or (C) hybrid?"
- **Open-ended is fine** when the answer genuinely needs exploration.
- **YAGNI ruthlessly**: if the user describes features that seem unnecessary, say so.
- **Stop after 3-5 questions**. If you need more, the scope is too large — suggest decomposition.

### Anti-Pattern: "This Is Too Simple To Need Questions"

Every feature goes through this process. A single config change, a new API endpoint, a UI toggle —
all of them. "Simple" features are where unexamined assumptions cause the most wasted work.
The Q&A can be brief (2-3 questions for truly simple features), but it MUST happen.

## Phase 3: Propose 2-3 Approaches

After understanding the requirements, present 2-3 different approaches:

### Format

```markdown
## Approach A: [Name] ⭐ Recommended
- **How**: [1-2 sentence description]
- **Pros**: [key advantages]
- **Cons**: [key disadvantages]
- **Effort**: [S/M/L]

## Approach B: [Name]
- **How**: [1-2 sentence description]
- **Pros**: ...
- **Cons**: ...
- **Effort**: ...

## My Recommendation: Approach [X]
[Reasoning — why this is the best fit for the stated requirements]
```

Lead with your recommendation and explain why. Let the user choose or combine.

## Phase 4: Present Design

Once approach is chosen, present the design **section by section** for incremental validation:

1. **Architecture**: Components, boundaries, data flow
2. **Interfaces**: Key APIs, data models, contracts
3. **Error handling**: Failure modes, recovery strategies
4. **Testing strategy**: What to test, how, at what level

After each section, ask: "Does this look right so far?"

If the user says no → go back, clarify, revise. Do NOT continue to the next section.

### Design Principles

- **Break into isolated units** with one clear purpose each
- **Well-defined interfaces** between units (can you understand a unit without reading its internals?)
- **Follow existing patterns** in the codebase — don't invent new conventions
- **Targeted improvements only** — fix what's in the way, don't refactor unrelated code

## Phase 5: Transition to Propose

When the design is approved:

```
Great, the design is clear. Let me generate the formal OpenSpec artifacts.

/opsx-propose <feature-name>
```

**Key**: The brainstormed understanding becomes CONTEXT for `/opsx-propose`.
The propose skill uses this context to generate more accurate specs.

### How Context Transfers

After brainstorming completes, the agent should carry these into `/opsx-propose`:

1. **Chosen approach**: Which of the 2-3 options was selected
2. **Key decisions**: Answers to the Socratic questions (scope, constraints, preferences)
3. **Design outline**: The section-by-section design that was approved
4. **Explicit out-of-scope**: What was deliberately excluded

The `/opsx-propose` DAG will use this to generate specs that match the brainstormed design,
rather than inferring requirements from a feature name alone.

## Anti-Patterns

| Anti-Pattern | Fix |
|-------------|-----|
| Skipping Q&A because "the user knows what they want" | They usually don't — that's why they need brainstorming |
| Asking 5 questions in one message | One at a time. Always. |
| Proposing only 1 approach | Always show 2-3 with trade-offs. Even if one is clearly better. |
| Presenting the entire design at once | Section by section, get approval after each. |
| Moving to propose before design approval | HARD GATE. Wait for explicit "looks good" or equivalent. |
| Brainstorming a feature that's actually 3 features | Flag scope. Suggest decomposition. Brainstorm one at a time. |

## Integration with Existing Workflow

```
Current:  /opsx-explore → /opsx-propose → /opsx-apply → /opsx-archive
Enhanced: /opsx-explore → /opsx-brainstorm → /opsx-propose → /opsx-apply → /opsx-archive
```

`/opsx-brainstorm` is the **mandatory bridge** between exploration and proposal.
It does NOT replace explore (which reads code) — it adds the design dialogue that propose lacks.
