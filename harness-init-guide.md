# Harness Engineering — 存量项目初始化（通用版）

> 适用于任何语言、任何 AI 编码工具。只讲思路和步骤，不绑定具体技术栈。

---

## 核心问题

AI Agent 在存量项目中反复失败：看到已有代码就提前交卷、环境跑不通但不自知、单测过了端到端不通、每次新对话重新摸索项目。

**一句话解法**: 别指望 Agent 记住规则——把规则变成自动化检查，Agent 根据报错改到通过为止。

---

## 四步法：盘点 → 分层 → 设卡 → 维护

每步做完向用户汇报，确认后再进下一步。

---

### Step 1：盘点

**目的**: 把 Agent 需要知道的东西写下来，放到它能读到的地方。

**盘五件事**:

| 维度 | 问什么 |
|------|--------|
| 构建命令 | 怎么装依赖、跑测试、跑 lint、部署？需要什么环境变量？ |
| 架构约束 | 哪些层不能互相依赖？模块边界在哪？有没有代码生成？ |
| 编码规范 | 命名风格、错误处理、日志格式、Response 结构 |
| 安全红线 | 哪些文件不能碰？凭证在哪（只标位置不复制值）？ |
| 隐性知识 | 团队口头约定、历史坑、"大家都知道但没写下来"的事 |

**怎么盘**: 扫构建文件、CI 配置、源码目录树（前 2-3 层）、README。安全文件只查文件名和行号，不读内容值。

**产出**: 一份根配置文件（你的 AI 工具能自动读取的那个文件），包含上面五个维度的关键信息。

**多仓库额外动作**: 对比各仓库的扫描结果，把所有不一致的地方（框架版本、包前缀、lint 配置等）明确标出来作为漂移警告。Agent 最致命的错误就是把 A 仓库的约定照搬到 B 仓库。

**不确定的信息**: 标注 `[TODO: confirm with team]`，收集后统一问用户，不猜。

**关键约束**:
- 根文件 ≤ 100 行 / < 32KB（太大会被截断或浪费 Token）
- 子仓库/子目录只写与根文件的差异，不重复
- 凭证只标 `文件名:行号 + 字段名`，绝不复制值

---

### Step 2：分层

**目的**: Agent 不需要一次加载所有知识。把"每次必须知道的"和"按需查阅的"分开。

**分两层**:

| 层级 | 放什么 | 加载时机 |
|------|--------|----------|
| 根文件 | 构建命令、架构红线、安全红线、文档索引 | 每次对话自动加载 |
| docs/ 目录 | 架构细节、开发指南、API 索引、架构决策记录(ADR) | Agent 需要时按需读取 |

**docs/ 建议包含**:
- **architecture.md** — 分层结构（ASCII 目录树）、基础设施集成、外部依赖、特殊模式
- **development-guide.md** — 环境搭建、本地开发、测试、迁移、部署
- **api-contracts.md** — API 规范索引（只做索引，不复制规范内容）
- **adr/** — 架构决策记录，每份 ≤ 40 行，必须有源码引用作为证据，5-8 份是合理范围

**瘦身根文件**: 把详细内容下沉到 docs/，根文件只留不变量（改了就出大事的东西）+ 文档索引表。

**质量要求**: 每个事实性声明必须有 `文件名:行号` 锚点指向实际源码。没有源码证据的声明标 TODO，不猜。

---

### Step 3：设卡

**目的**: 把文档里写的规则变成可执行的自动化检查。这是最关键的一步。

> Agent 不需要"记住"任何规则。它只需要跑检查命令，然后根据报错改到通过。

**五步走**:

1. **盘点已有检查** — 项目已经有哪些 lint / test / CI 检查？不要重复建设
2. **识别高频违规** — Agent（和人）最容易犯什么错？从 Code Review 反复出现的评论中提取
3. **新增自动化检查** — 按低摩擦优先的顺序添加：
   - **架构分层检查** — 禁止跨层依赖（底层不能 import 上层）
   - **代码风格 / Linter** — 统一风格，捕获常见错误
   - **类型检查**（如语言支持） — 渐进式启用
   - **覆盖率阈值** — 设略低于当前实际值，防止退步
4. **挂到 CI** — 确保每个 PR / merge 都跑这组检查
5. **挂到本地** — pre-commit hook + AI 工具的本地配置，提交前拦截

**棕地项目棘轮策略**（核心！）:

存量项目不可能一步到位。正确做法是**冻结-棘轮**：
1. 首次运行，记录（或冻结）存量违规
2. 只有**新增**违规才阻断构建
3. 随着团队修复存量问题，基线自动收紧

绝不要在棕地项目直接开严格模式——几百个存量错误会直接炸掉构建，团队会放弃整个方案。

**在根配置文件中更新**: 加一节"质量门禁"，写明检查命令。Agent 每次改代码后跑这组命令，不过就不提交。

---

### Step 4：维护

**目的**: 让文档活着。过期的记忆比没有记忆更危险——Agent 读到错误的文档，产出的就是幻觉。

| 机制 | 做什么 | 频率 |
|------|--------|------|
| Doc-gardening | 扫描文档中的 `文件:行号` 引用是否仍正确，事实性声明是否与代码一致 | 每周 |
| 规则回顾 | 违规率在下降吗？棘轮能收紧吗？阈值能提高吗？无效规则该删了吗？ | 每两周 |
| 根文件瘦身 | 膨胀超过 100 行了吗？信息该下沉到 docs/ 了吗？ | 每月 |

**自动修复规则**: 只改 `.md` 文件中有 grep 验证的事实性纠正（如行号偏移）。需要判断的、任何代码文件——不碰。

---

## 常见误区

| 误区 | 正确做法 |
|------|----------|
| 5000 行的根配置文件 | ≤ 100 行索引，详细内容放 docs/ |
| 只写文档不设自动检查 | 关键规则必须变成可执行 lint / test |
| Agent 自己验证自己的产出 | 生成与评估必须分离（Agent 写 → Linter 验） |
| 一次性写好不维护 | 文档是活的，需要持续 gardening |
| 棕地项目直接全量 strict | 冻结-棘轮法：先宽松启动，逐步收紧 |
| 子仓库复制根文件内容 | 只写差异 |
| 文档里写凭证值 | 只写 `文件名:行号` + 字段名 |

## 时间线参考

```
Day 1：盘点
  扫描仓库 → 生成根配置文件 → 团队补充 → TODO 确认

Day 2：分层
  根文件瘦身为索引 → 创建 docs/ → 写 ADR → 加文档索引

Day 3：设卡
  盘点现有检查 → 加架构分层检查 + Linter + 覆盖率阈值 → 配 pre-commit → 更新根文件

持续：维护
  每周 gardening · 每两周规则回顾 · 每月根文件瘦身
```
